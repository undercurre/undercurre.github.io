跨域总论

什么是同源策略？

同源策略是指只有具有相同源的页面才能够共享数据，比如 cookie，同源是指页面具有相同的协议、域名、端口号，有一项不同就不是同源。 有同源策略能够保证 web 网页的安全性。

跨域一定会在非同源的情况下发生吗？

是的，只要非同源情况下，跨域就会发生，不仅仅发生在 Ajax 请求，img 资源和 script 请求也会发生跨域。

所有的跨域请求都会出现问题吗？

不是的，脱离的浏览器范围不存在跨域问题的，因为这一切的规则都是浏览器出于安全考虑搞出来的，并且只要在响应到达浏览器后的跨域校验通过了就不会引发跨域问题。

跨域请求的情况下浏览器还会发出请求吗？服务器能收到请求吗？

浏览器会正常发出请求，服务器也会收到，请求的跨域限制发生在请求响应阶段，服务器响应了过后，浏览器会对这个响应进行校验，因为他知道这是跨域，如果说不跨域的话，那就没有任何校验，把这个结果放行，但是由于是跨域，所以说他有一套安全的闸门。来看一下校验会产生两种结果，一种是校验通过，放行响应，关键是校验不通过的情况下，如果说校验不能通过的话，它就会引发一个错误，那么这个错误就是我们平时说的跨域问题。

跨域的最好解决方案是什么?

CORS 是最正统的解决方案，也是官方推荐使用的方案。

跨域请求的到达校验是怎么样的？

校验的规则就叫做 CORS 规则，其实它的整体原则很简单，就是让服务器来表个态，服务器觉得都是自己人没问题，那服务器都表示允许了，浏览器也不好说什么，他就校验通过了，就这么简单，而服务器没有表示不说话或者是明确表示拒绝，那校验就不通过，所以它的原则就是那么个原则。

CORS 方案是什么？

需要服务器返回响应头 Access-Control-Allow-Origin 符合当前页面源

CORS 来解决跨域问题的话，主要依赖谁？

主要靠服务器，所以你就必须要确保服务器能够得到控制的情况下使用 CORS 方案。

什么是预检请求？

CORS 方案它把请求分为两类，一种是简单请求，一种是预检请求。预检请求一般是请求头和 content-type 被修改过的请求，因为浏览器觉得是高危请求所以在发起请求前先发送 method 为 options 的预检请求到服务器确认是否是被允许的源，再发送真正的请求，这个真实请求就是一个普通请求。

分析一下为什么有时候遇到跨域情况，上传图片成功，但是提交表单则出问题？

上传图片用的是 content type 是用 multiple form data ，上传图片很有可能是一个简单请求，说明服务器是允许简单请求的。当然是提交普通表单，那么这可能是一个 Ajax，而我们通常提交普通表单的时候，用 JSON 格式发过去的，content type 是用 application/json，就变成了一个预检请求了,而预检请求的时候遇到了跨域问题，就有可能是服务器它支持简单请求，不支持预检请求导致的。

JSONP 是什么？处理跨域问题已经存在 CORS 方案，为何还需要 JSONP 方案呢？

JSONP 诞生于没有 CORS 的年代，通过绕过 Ajax 来实现跨域请求。开发者会创建这么一个 script 元素，然后用 SRC 给它填入跨域的地址，所以说相当于是用标签的形式来进行跨域请求，由于浏览器对标签的跨域的限制较小，所以说能够正常的请求正常的响应，并且能够拿到这个响应结果，而且还能把这个响应结果当成一个 JS 来执行。这就要求我们服务器得给我响应一段 JS 代码，这段 JS 代码里边就是调用我之前准备好的那个函数，然后把我们需要的数据作为参数给我传过来，运行这个函数。于是就收到并解析出响应结果。

JSONP 来解决跨域问题的话，需要满足什么条件？

必须要确保服务器能够得到控制

在目标服务器不可控的情况下，如何解决跨域问题？

使用代理服务器，代理该请求，使该请求脱离浏览器。

JSONP 是否适用于任何类型的跨域请求？

不是的，JSONP 只适用于 GET 请求，因为 script 元素只能发出 GET 请求
