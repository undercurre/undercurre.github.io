ESlint 和 Prettier 冲突解决

相信大家都想得到一个很重要的问题，就是上述两者可能会存在冲突，当 ESLint 的规则和 Prettier 的规则相冲突时，就会发现一个尴尬的问题，用其中一种来格式化代码，另一种就会报错。

那怎么解决呢？

eslint-plugin-prettier 和 eslint-config-prettier 会帮我们解决这个问题。

其中 eslint-config-prettier 会把 Eslint 里和 Prettier 有冲突的规则关掉，eslint-plugin-prettier 会将 Prettier 里的规则设置到 Eslint 里面，通过这两个插件的配合，就完成了 Eslint 和 Prettier 规则的合并，其中冲突的规则以 Prettier 里的规则为准。

然而 Prettier 只会管理代码的格式化不会理解 css 代码的规则和语法

styleLint可以帮你解决这个问题

安装: pnpm install –save-dev stylelint

相关插件:
stylelint-config-standard 是 stylelint 的推荐配置

stylelint-config-standard 是 stylelint 推荐配置的 scss 版

stylelint-order 是 CSS 属性排序插件(先写定位，再写盒模型，再写内容区样式，最后写 CSS3 相关属性)。

工程化命令：npm pkg set scripts.stylelint="stylelint \"./**/*.{css,scss,vue,html}\" --fix"

每一次都要手动格式化是很痛苦的事情，要全局格式化更痛苦！！！

lint-staged 是一个在 git 暂存文件上运行 linters 的工具,使用它你能如愿解决烦恼。

安装：pnpm add lint-staged -D

快捷使用: npx mrm lint-staged

并且npx mrm lint-staged会自动配置好 husky 来拦截 git 提交钩子来检查

但是到这里也只是对代码的检查，我们还需要对提交的消息记录做规范

commitlint 用于检查 git 提交消息是否符合指定的规范和风格。

commitizen 是用来格式化 git commit message 的工具，它提供了一种问询式的方式去获取所需的提交信息。

@commitlint/cli 提供校验的执行入口 commitlint

创建 commit-msg 钩子执行 消息校验

windows 执行npx husky add .husky/commit-msg "npx --no-install commitlint --edit $1"

linux 执行npx husky add .husky/commit-msg 'npx --no-install commitlint --edit "$1"'
